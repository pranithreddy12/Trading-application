91: Section III — ATLAS System Architecture
92: ────────────────────────────────────────────────────────────────────────────────
93: 3.1  Architecture Philosophy
94: ATLAS is built on five architectural principles that govern every design decision in this document.
96: Principle 1 — Agents Communicate Across Levels, Not Just Within
97: Every agent has both vertical outputs (to the next layer in the pipeline) and horizontal outputs (to agents in other layers that benefit from its signal). A Data Collector does not merely hand data to a Feature Engineer — it simultaneously signals Strategy Ideators when anomalies appear, and alerts Risk Controllers when data feeds degrade. This cross-level mesh creates compound intelligence: the output of the whole system exceeds the sum of individual agents.
99: Principle 2 — Agents Spawn Agents
100: The Meta-Orchestrator does not manage a fixed roster. When backtest volume exceeds capacity, it spawns additional Backtest Runner instances. When a new asset class is added, it spawns the corresponding Data Collectors, Feature Engineers, and Ideators automatically. The system scales by multiplication, not by manual configuration.
102: Principle 3 — The Data Backend Is a First-Class Product
103: Every data point ATLAS touches — raw market data, enhanced features, strategy performance, agent behavior, validation results, execution quality, scout intelligence — is stored, indexed, and available for ML training. The data backend is not an afterthought; it is one of the system's three primary outputs alongside trading P&L and strategy generation. Within 12 months, this database will be the largest proprietary repository of enhanced, quality-verified trading data and strategy performance data in existence. It is the fuel for machine learning models that will compound ATLAS's advantage indefinitely.
105: Principle 4 — Every Layer Self-Improves
106: Self-improvement is not limited to the Meta-Learning layer. Every layer feeds performance data back to its own agents and to the Meta-Learner. Data Collectors track their own latency and accuracy. Ideators track which hypothesis patterns produce validated strategies. Validators track their false-positive and false-negative rates. Execution agents track slippage and optimize routing. The entire system is a learning organism, not a static pipeline.
108: Principle 5 — Security Through Lifecycle Management
109: Agents that operate on trader laptops (the Fidelity Monitoring Agents) follow strict lifecycle rules: they activate at market open, monitor during trading hours, and are killed at market close. Only the cloud-based data feed agent persists overnight, collecting news and pre-market data to generate opening hypotheses. This architecture ensures that no sensitive trading data persists on local machines outside of active trading hours.

169: 22 agents. The sensory system. Everything ATLAS knows about the world flows through this layer. The quality, speed, and granularity of data ingestion determines the ceiling of every other layer.
170: 4.1  Design Imperative: Level 2 & Order Flow First
171: ATLAS is not built on delayed OHLCV bars. The primary data sources are Level 2 market data (full order book depth) and order flow (time & sales, trade prints, volume at price). These are the fastest and most accurate representations of market state available to non-HFT participants. All other data sources (OHLCV, news, sentiment, macro) are supplementary.
172: ⚠  Level 2 and order flow data generate significantly higher data volumes than OHLCV. TimescaleDB hypertable partitioning and Redis hot-layer caching are essential to prevent the database from becoming the bottleneck. Size the infrastructure accordingly — see Section XIV.

This agent does not compute known indicators — it runs unsupervised anomaly detection and statistical pattern recognition across the full enriched feature set, looking for structures that no predefined indicator would capture. These are fed to the Strategy Ideators as raw alpha signals alongside traditional features. This is the 'enhanced data feeding pattern recognition for entries and exits' you specified.
178: 4.5  Data Quality & Monitoring (4 agents)
179: Layer 1 Total: 22 agents. Scaling: +1 Collector per new data source, +1 Feature Engineer per new feature category, +1 Pattern Recognition variant per asset class.
180: 4.6  The Enhanced Data Backend (ML Training Store)
181: Every data point that enters L1 — raw, enriched, and enhanced — is stored in a purpose-built backend designed for future ML model training. This is not a log archive. It is a structured, indexed, queryable data warehouse that will become the most valuable asset ATLAS produces after trading P&L itself.
182: Raw market data: Level 2 snapshots, order flow, OHLCV (all timeframes), options chains, futures term structure — partitioned by date and symbol in TimescaleDB hypertables.
183: Enhanced features: the full 400+ feature library, computed and stored at every timestep, in PostgreSQL + Redis.
184: Strategy performance: every strategy hypothesis, its code, backtest results, validation outcomes, paper/live P&L — all linked to the market conditions at the time. This is the training data for 'what strategies work in what regimes.'
185: Agent behavior: every agent action, decision, and error — linked to outcomes. This is the training data for 'what agent behaviors correlate with profitability.'
186: Scout intelligence: every external hypothesis sourced, its origin, its validation outcome. This is the training data for 'which external sources produce real alpha.'
187: Execution data: every order, fill, slippage measurement, latency — linked to market conditions at execution time. This is the training data for 'optimal execution under different conditions.'
189: Schema design: star schema with fact tables (trades, signals, features, agent_events) and dimension tables (strategies, agents, instruments, regimes). Optimized for analytical queries and batch ML training jobs. See Appendix D for full schema.
190: ⚠  Data volume estimate: with Level 2 + order flow for 200+ US equities, ATLAS will generate approximately 50-100 GB of raw data per trading day. TimescaleDB compression (typically 10-20x) reduces storage needs, but the infrastructure must be sized accordingly. Budget for 2 TB storage in Month 1, scaling to 10 TB by Month 6.

193: Section V — Layer 2: Strategy Generation Engine
194: ────────────────────────────────────────────────────────────────────────────────
195: 14 agents. The creative engine. Where alpha is born. This is the layer that must 'think like a mad scientist' — aggressive exploration of every possible edge, combined with mathematical rigor in risk parameterization. The philosophy: everything is possible until proven otherwise by data. But every hypothesis is tested with uncompromising statistical discipline before a single dollar is deployed.
196: 5.1  Strategy Ideators (7 agents)
197: Each Ideator specializes in one asset class or strategy archetype, receives the full feature library plus Pattern Recognition signals from L1, external hypotheses from the Scout Network, and failure/success patterns from the Meta-Learner.
198: The 'Mad Scientist' Protocol
199: Each Ideator agent is configured with an aggressive exploration mandate:
200: Temperature: set high (0.9-1.0) for initial hypothesis generation — explore widely.
201: Constraint: every hypothesis must include explicit mathematical risk parameters (stop loss, position size formula, max drawdown limit, regime filter). Creativity in signal; discipline in risk.
202: Volume target: each Ideator produces 10-20 hypotheses per day. Across 7 Ideators, that is 70-140 hypotheses daily, 500-1,000 weekly.
203: Mutation integration: every 5th hypothesis is a deliberate mutation of a previously validated strategy — change one parameter, one filter, one universe constraint, and re-test.
204: Failure analysis: every Ideator receives a daily feed from L7 of the top 10 failure patterns from yesterday's rejected strategies. The prompt: 'Do not generate strategies that would fail for these reasons.'
205: ▸  The 'mad scientist' approach works because the cost of generating a bad hypothesis is near-zero (one Claude API call, ~$0.01-0.05). The cost of missing a profitable strategy is infinite opportunity cost. Generate aggressively, validate ruthlessly.

208: 5.4  Generation Volume Targets
209: Validation pass rate is intentionally 10-15%. Most generated strategies SHOULD fail. The value is in the survivors. Volume is the strategy.

236: Section X — Layer 6: Copy Trading & Fidelity Monitoring
237: ────────────────────────────────────────────────────────────────────────────────
238: 7 agents. Mirror human trader edge with machine precision. The Fidelity Monitoring Agent is a new addition — a specialized agent that lives on trader laptops during market hours, monitors every action, and is killed at market close for security.
239: Fidelity Monitoring Agent — Detailed Specification
240: The Fidelity Monitoring Agent (L6-FM-01) is the manual trading side's connection to ATLAS. One instance runs on each Fidelity trading laptop. Its responsibilities:
241: Capture every order entry, fill, modification, and cancellation in real-time
242: Cross-reference each trade against ATLAS's own data stream (Level 2 + order flow from L1) to identify whether the trade aligns with any ATLAS strategy signals
243: Log all trading activity to the cloud backend for the Enhanced Data Store (feeding ML training)
244: Monitor for compliance: position size limits, drawdown limits, unauthorized symbol trading
245: Generate post-session trade journal automatically: every trade with entry/exit times, P&L, notes on market conditions
246: Lifecycle Management (Security Protocol)
247: PRE-MARKET: The Meta-Orchestrator spawns L6-FM-01 instances at 9:25 AM ET (5 minutes before market open). Each instance connects to its assigned laptop and begins monitoring.
248: MARKET HOURS: Full monitoring active. All data streams live.
249: POST-MARKET: At 4:05 PM ET (5 minutes after close), each L6-FM-01 instance uploads final session data to cloud, then self-terminates. No monitoring agent runs on any laptop outside market hours.
250: OVERNIGHT: Only L6-FM-02 (cloud agent) persists. It collects overnight news, pre-market data, and generates the opening hypothesis document for the next trading day.
251: NEXT MORNING: Meta-Orchestrator spawns fresh L6-FM-01 instances at 9:25 AM ET. Clean lifecycle. No stale state.
252: ⚠  This lifecycle ensures no sensitive trading credentials or real-time position data persists on local machines outside market hours. All persistent data lives in the encrypted cloud backend only.

255: Section XI — Layer 7: Meta-Learning & Self-Improvement
256: ────────────────────────────────────────────────────────────────────────────────
257: 10 agents. The brain. The layer that makes ATLAS better every day. This is the layer you make immortal — the compounding advantage that separates ATLAS from every static system.
258: The Self-Improvement Compound Loop
259: Every day ATLAS operates, it collects data that makes it better the next day. This is not aspirational — it is architectural. The specific feedback loops:
260: Loop 1: Generation → Validation → Meta-Learner → Generation. Ideators learn which hypothesis patterns produce validated strategies. Daily update.
261: Loop 2: Live Performance → Meta-Learner → Risk Adjustment. Risk parameters auto-adjust based on actual vs expected performance.
262: Loop 3: Feature Importance → Feature Engineers → Ideators. The features that actually predict profitability get prioritized. Features that don't get deprecated.
263: Loop 4: Agent Behavior → Agent Evaluator → Meta-Orchestrator. Agents that produce better outcomes get cloned. Agents that underperform get reconfigured or killed.
264: Loop 5: Scout Reliability → Scout Prioritization. Sources that produce strategies that actually validate get higher priority in the External Hypothesis Queue.
265: Loop 6: Execution Quality → Slippage Model → Backtesting. Real slippage data feeds back to make backtests more realistic, which makes validation more accurate, which makes live deployment less surprising.
267: ▸  After 30 days of operation, ATLAS has 30 days of compound learning. After 90 days, no competitor starting from zero can catch up. After 180 days, the data moat is permanent. This is not about being smart on Day 1 — it is about being unchallengeable by Day 180.
